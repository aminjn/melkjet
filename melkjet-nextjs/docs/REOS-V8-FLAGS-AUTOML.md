# REOS v8 · Feature Flags + AutoML — گزارشِ ممیزی

> پاسخ به دو موردِ کلیدیِ ممیز: (۱) **Feature Flags** — قانونِ نگهداری‌پذیری که خواسته بود «هر لایه مستقل و
> قابلِ خاموش/روشن باشد»، و (۲) **AI Experiment Platform** — تنها موردی که ممیز به‌عنوانِ CTO اضافه می‌کرد:
> champion/challenger که **خودکار** بهترین مدل را انتخاب و جابه‌جا کند، بدونِ دخالتِ انسان.
>
> **تاریخ:** 2026-07-07 · قابلِ راستی‌آزمایی با **۴۲۷ تست** (۲۰۰ واحد + ۲۲۷ روی PostgreSQLِ واقعی).

---

## ۱) Feature Flags — `flags.ts`
هر لایه‌ی REOS اکنون **مستقل** قابلِ کنترل است، بدونِ دیپلوی:
- **enabled** (روشن/خاموش) · **rolloutPct** (عرضهٔ تدریجی ۰..۱۰۰) · محدودسازی به **cities/plans/roles**.
- **سطل‌بندیِ قطعی** (`bucketOf` با SHA1): همان کاربر همیشه یک نتیجه می‌گیرد — عرضهٔ ۵۰٪ پایدار و تکرارپذیر است.
- فلگ‌های پیش‌فرض برای لایه‌ها: `dominance · economy · community · wallet · automl` (پیش‌فرض روشن ۱۰۰٪ تا رفتارِ فعلی حفظ شود).
- مثال‌های خواسته‌شدهٔ ممیز، همه ممکن: «Dominance فقط تهران» (city scope)، «Community فقط ۱۰٪» (rollout)، «Wallet فقط Pro» (plan scope).

**اعمالِ واقعی (نه دکوری):** روت‌های `/api/reos/territory`، `/economy`، `/community` در ابتدای GET فلگِ خود را
می‌سنجند؛ اگر برای این کاربر روشن نباشد `{disabled:true}` برمی‌گردانند و **کارتِ UI خود را پنهان می‌کند**.
> مدیریت در `/reos-admin` → «🚩 فلگ‌های ویژگی»: سوییچِ روشن/خاموش + درصدِ عرضه + محدودسازی به شهر.

## ۲) AutoML — پلتفرمِ آزمایشِ خودکار — `automl.ts`
دقیقاً همان چیزی که ممیز به‌عنوان تنها افزودهٔ CTO خواست:
- **`shouldPromote`** (هستهٔ خالص): چالش‌گر فقط وقتی جای قهرمان را می‌گیرد که (الف) با **حاشیهٔ مطمئن** (config
  `promoteMargin`, پیش‌فرض AUC +۰٫۰۲) بهتر باشد و (ب) **نمونهٔ کافی** (`minSamples`, پیش‌فرض ۱۰۰) داشته باشد.
  → جلوی ارتقای مدلِ خوش‌شانسِ کم‌داده گرفته می‌شود.
- **`autoPromote(name)`**: قهرمان و بهترین چالش‌گر را می‌سنجد و در صورتِ برتری، **خودکار** `promote` می‌کند
  (قهرمانِ قبلی retired می‌شود — rolloutِ ایمن). خروجی شفاف است (چه شد و چرا).
- **کرون:** هر ۶ ساعت بعد از آموزشِ مدل‌ها، `runAutoML(['engage','lead'])` اجرا می‌شود → سیستم بدونِ دخالتِ
  انسان کیفیتِ خود را بهبود می‌دهد. «پلتفرم یاد می‌گیرد بهتر تصمیم بگیرد، نه فقط مدل‌ها از داده آموزش ببینند.»
> نمایش/اجرا در `/reos-admin` → «🤖 AutoML»: قهرمان/چالش‌گرِ هر مدل + برچسبِ «آمادهٔ ارتقا» + دکمهٔ اجرای دستی.

## ۳) کنترلِ سوپرادمین (همه واقعی)
`/reos-admin` → مرکزِ کنترل: بخشِ **AutoML** در config (فعال/حاشیهٔ ارتقا/حداقلِ نمونه) + مدیریتِ **فلگ‌ها** +
دکمهٔ «اجرای دورِ AutoML». همه به موتورِ زنده وصل‌اند.

## ۴) رعایتِ توصیهٔ معماریِ ممیز
ممیز هشدار داد لایه‌ها بیش از حد پیچیده نشوند و هرکدام مستقل/قابل‌تست/قابل‌خاموش باشند. Feature Flags **دقیقاً همین**
را می‌دهد: هر لایه پشتِ فلگِ خود، مستقل قابلِ عرضه/آزمایش/بازگردانی. به همین دلیل فقط این دو مورد (از ۱۰ پیشنهاد)
ساخته شد — نه همه‌چیز هم‌زمان.

## ۵) راستی‌آزمایی
```
node --import ./scripts/reos-loader.mjs scripts/reos-test.mjs                 # ۲۰۰ سبز (+۱۹: flags/automl خالص)
DATABASE_URL=… node --import ./scripts/reos-loader.mjs scripts/reos-store-test.mjs   # ۲۲۷ سبز (+۱۲: flag store + promote روی رجیستریِ واقعی)
```
- **flags واقعی؟** تستِ «rollout 0 → off»، «city scope gates»، «deterministic bucket».
- **AutoML واقعی؟** تستِ «promotes clearly-better»، «does NOT promote high-AUC with too-few-samples»، «no promotion when stable» روی Model Registryِ واقعی.
- **صادقانه:** `shouldPromote` فرمولِ قطعیِ توضیح‌پذیر است (نه جعبهٔ سیاه)؛ ورودی‌اش متریکِ واقعیِ مدل‌های آموزش‌دیده است.
